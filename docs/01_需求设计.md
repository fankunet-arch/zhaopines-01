# 01 · 需求设计（定稿 v1.0）

> 描述「做什么」与「业务规则」，不含技术实现（见 `02_技术架构.md`）。

---

## 1. 项目概述

### 1.1 定位
极简的**信息发布型招聘平台**，形态接近"分类信息墙/电子公告板"，而非全功能招聘系统。核心体验是"贴一条信息就走"，不做简历库、在线沟通、智能匹配等重功能。

### 1.2 目标用户
西班牙华人**低端服务业**群体：餐厅（厨师、跑堂）、百元店、美甲店等小商户（雇主侧）及其求职者。共同特征：时间少、不耐烦复杂操作、以手机为主。**一切设计以"降低操作成本"为最高优先级。**

### 1.3 产品形态
纯中文界面、移动优先、底部附西语合规声明；分**前台 / 用户后台 / 管理员后台**三端；信息分**招聘 / 求职**两类。

### 1.4 差异化优势（凭什么赢）
对手是欧浪等"华人版综合信息站"。本站不比全，比**专、快、干净、可信、懂行**：
- **窄**：只做这个圈子的招聘，字段/分类/文案全为该人群定制。
- **快**：免注册可发、字段极简，30 秒发完。
- **新陈代谢**：到期下架 + 失效治理，信息永远是活的，对抗老站的僵尸帖。
- **可信**（最大护城河）：举报做实、失效标记、合规姿态，对抗华人圈假招聘/诈骗。
- **懂行**：本地化语气，自己人感。

## 2. 范围界定

**本期要做**：招聘/求职信息发布、浏览、筛选、搜索；游客免注册发布 + 注册用户管理；举报 + 邮件提醒；联系方式点击才显示；轻量防机器人/防爬；置顶券；失效治理；用户后台、管理员后台。

**本期明确不做**（防需求膨胀）：简历库、站内 IM、薪资/工时字段、智能匹配/推荐、游客编辑密码机制、游客帖认领、手机短信验证码。

## 3. 角色与权限

| 角色 | 进入方式 | 定位 |
|------|----------|------|
| 游客 | 无需登录 | 保有最大信息源：能浏览、能发布，但发了不能改 |
| 注册用户 | Google 登录 | 要控制权就注册：能发布并编辑/删除/顶帖/用券 |
| 管理员 | Google 登录 + 邮箱白名单 | 审核类别、处理举报、删信息、发券、配置参数 |

### 权限矩阵

| 操作 | 游客 | 注册用户 | 管理员 |
|------|:----:|:--------:|:------:|
| 浏览/筛选/搜索 | ✅ | ✅ | ✅ |
| 查看联系方式（点击显示） | ✅ | ✅ | ✅ |
| 发布信息 | ✅ | ✅ | ✅ |
| 编辑/删除自己的信息 | ❌ | ✅ | ✅ |
| 顶帖（7天内刷新置前） | ❌ | ✅ | — |
| 使用置顶券 | ❌ | ✅ | — |
| 标记信息已失效 | ✅ | ✅ | — |
| 主动下架自己的帖（已招到/已找到） | ❌ | ✅ | ✅ |
| 举报信息 | ✅ | ✅ | — |
| 删除任意信息 | ❌ | ❌ | ✅ |
| 审核新职位类别 | ❌ | ❌ | ✅ |
| 处理举报 | ❌ | ❌ | ✅ |
| 生成/管理置顶券 | ❌ | ❌ | ✅ |
| 配置系统参数 | ❌ | ❌ | ✅ |

### 关键边界
- 游客帖一经发布即"泼出去的水"：本人不可改/删，只能等 30 天到期或被管理员删。
- **游客帖不可认领**：游客后来注册也无法把先前的游客帖转为可编辑（无法证明归属）。想要可管理的帖子，注册后重发。
- 注册的唯一价值即"获得对自己信息的控制权"。

## 4. 核心业务规则

### 4.1 有效期 + 数量保底
- 信息发布满 **30 天**自动下架（不在前台显示），**数据库保留**。
- **保底最新 N 条**：前台永远至少保留最新 N 条；在线不足 N 条时把最近下架的捞回补满，保证墙不空。N 后台可调；**招聘、求职各自独立计算**。

### 4.2 重复发布与频率
- 同电话（同人）+ 同内容：**每天限发 1 条**。
- **7 天内**再发相同信息：不新增记录，把原记录**顶帖**（更新时间戳置前）。仅注册用户可顶帖。

### 4.3 注册与登录
- **彻底零成本**：注册/登录仅 **Google**；无账号者走游客发布。
- **不做手机短信验证码**（无零成本方案）。
- **管理员也走 Google + 邮箱白名单**：Google 验证通过且邮箱在白名单内，双重满足才授权。
- **全站零密码**：系统不存任何密码/哈希。
- **安全前提**：管理员的 Google 账号必须开启 2FA。

### 4.4 游客发布机制
游客不可编辑，因此**不设编辑密码机制**（无需生成/保管/找回，系统简化，也无"密码丢了改不了"的痛点）。

### 4.5 排序
置顶（券/付费）最前 → 其余按发布/顶帖时间倒序。

### 4.6 字段
| 字段 | 招聘 | 求职 | 说明 |
|------|:----:|:----:|------|
| 信息内容 | 必填 | 必填 | 自由文本（限长） |
| 联系人/姓名 | 必填 | 必填 | |
| 联系电话 | 必填 | 必填 | 点击才显示 |
| 微信号 | 选填 | 选填 | 处理同电话：点击才显示 + 字段名混淆 |
| 所在地区 | 必填 | 必填 | 固定两级下拉 |
| 职位类别 | 必填 | 必填 | 固定下拉 + 可录入待审核 |

不做薪资/工时。

### 4.7 地区
覆盖全西班牙，**固定两级下拉（大区/市，到市级为止）**，不允许自由输入。种子数据见 `seed_regions.sql`。

### 4.8 职位类别（审核隔离）
前台固定下拉；允许用户打字录入新类别 → 进**待审核类别表**（与正式表物理隔离，防系统错乱）→ 管理员审核通过才并入正式表。初始类别见 `seed_categories.sql`。

### 4.9 举报与邮件提醒
- 任何人可举报；产生举报时**向管理员发提醒邮件**，收件邮箱后台可配（可多个）。
- 防刷爆双护：① **举报频率限制**（同 IP 对同帖短时间内重复举报不重复计数）；② **邮件合并降频**（同帖 1 小时内最多 1 封，写明"累计被举报 N 次"）。阈值后台可调。

### 4.10 置顶券（兑换码模式）
- **仅对注册用户**。
- 管理员生成兑换码（随机不可枚举），线下/微信发给用户，用户后台输入码兑换入账。
- 发券设两参数：**券有效期**（未兑换即作废）、**置顶时长**（用后置顶天数）。
- 一张券用于一条自己的帖子；置顶到期自动回落普通排序。**置顶不延长帖子 30 天寿命**。
- **台账与防丢**：可查每张券状态/归属/所用帖子；用户丢码可由管理员作废旧码、重发新码。
- **默认不防转赠**（赠送性质、无现金价值，转赠等于为平台带来活跃）。

### 4.11 置顶位上限（后台可调）
同时置顶条数设上限。**上限是"准入闸"非"裁决者"**：仅在新置顶进场时判断（在场数<上限才放行），**绝不踢已在场的**。调小上限（如 6→5）时存量活到各自过期、期间不进新的，回落到上限下才再放行。

### 4.12 信息失效治理
- **浏览者标记"已失效"**（空号/岗位没了）：累计达阈值（后台可调）自动降权或提醒管理员；独立于举报（善意纠错）。
- **发布者主动下架**：注册用户标记"已招到/已找到"提前下架。

### 4.13 正文夹带联系方式封堵
发布时自动检测"信息内容"中的电话/微信格式，命中则提示改填联系方式栏或屏蔽，防止绕过"点击才显示 + 防爬"。

### 4.14 联系方式查看次数统计
- 取号（点击显示电话/微信）时**分别计数**：电话查看数、微信查看数各自独立累计，可看出用户更倾向来电还是加微信。
- **前台公开显示**每条信息的查看次数（形成热度心理，促活跃）。
- **防刷**：同 IP 对同一帖同一类型短时间内的重复取号**不重复计数**（复用取号去重逻辑）。
- 前台展示累计数；管理员后台可基于取号明细算去重"N 人查看"及电话/微信偏好。

## 5. 功能模块

### 5.1 前台
板块切换（招聘/求职）；类别+地区筛选；关键词搜索；信息列表（按 §4.5 排序）；详情页（联系方式点击才显示，显示后支持一键拨号/复制微信，**公开显示电话/微信各自的查看次数**）；发布入口（校验手机号格式、封堵正文夹带）；举报入口；标记"已失效"入口；分享到微信/复制链接。

### 5.2 用户后台（注册用户）
我的发布；编辑/删除；顶帖；我的置顶券（兑换码兑换、查看、对某帖使用）；主动下架。

### 5.3 管理员后台
信息管理（查看/删除/下架任意信息）；职位类别审核；举报处理；举报邮件提醒（频率限制+合并降频）；收件邮箱配置；失效标记处理；置顶券管理（生成/台账/作废/重发）；可疑信息提示（后台高亮，内容仍正常显示，人工决定删否）；数据看板（新增/分布/活跃，建议简版先行）；参数配置（保底 N、置顶位上限、各类频率与时间窗、字段名轮换、失效阈值、敏感词库等）。

## 7. 防护需求（轻量防护：尽力而为，非安全保证）

> **定位声明**：以下手段旨在**提高骚扰/采集成本**，非杜绝。系统不对被爬/被刷做安全承诺。

### 7.1 防机器人群发
验证码 + 蜜罐字段（隐藏框被填即判机器人）+ 频率限制；**不人工前置审核**，可疑信息照常显示、仅后台高亮，人工决定删否。

**可疑判定初始规则（后台可调）**：① 同 IP 高频发布；② 触发蜜罐；③ 内容含垃圾特征（多外链、博彩/贷款等）；④ 同内容跨账号/IP 反复；⑤ 提交速度异常（疑似脚本）；⑥ 同一电话被大量不同"发布者"使用。

### 7.2 防爬虫采集
联系方式默认不明文、**点击时才请求取号接口**（可加频率限制）；**正文夹带封堵**（§4.13）；**字段名混淆**（无意义名如 `f_03`，可周期轮换）。不做图片化电话、单 IP 严格限量（体验优先，避免误伤共用网络的真实用户）。

### 7.3 违法招聘敏感词提示（合规）
轻量敏感词库，命中只做**后台标记/提示，不强制拦截**。

## 8. 非功能需求

- **语言**：纯中文界面 + 底部西语合规声明。
- **合规（上线前必备）**：隐私政策 + Cookie 提示（GDPR 强制，中/西双语）；个人数据最小化收集（能不存邮箱就不存）。
- **安全**：数据库内部主键自增**永不对外暴露**，对外用不可枚举随机标识；兑换码亦随机不可猜；机密放 `app/config` 不入库。
- **运维（上线前必备）**：数据库定期自动备份。
- **终端**：移动优先，兼顾桌面。
- **SEO**：服务端渲染、每条信息独立 URL、语义化标题、JobPosting 结构化数据、sitemap（详见 `02`）。
- **数据保留**：下架信息数据库保留，不物理删除（除主动/管理员删除）。
- **可配置**：保底 N、置顶位上限、各类阈值、字段名轮换、可疑/失效判定、敏感词库等均后台可调。
